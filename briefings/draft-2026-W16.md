This week, the attack surface for agentic AI collapsed faster than most security teams can track. Across every layer — open-source builders, protocol design, developer tooling, and enterprise platforms — attackers and researchers found the same thing: AI agents are running with too much trust, too much access, and almost no meaningful guardrails.

Start with Flowise. CVE-2025-59528 is a CVSS 10.0 remote code execution flaw in the CustomMCP node of this widely-deployed open-source AI agent builder. Over twelve thousand internet-exposed instances are at risk right now, and active exploitation is confirmed. The vulnerability requires no authentication. An attacker passes unsanitised input through an MCP server configuration field and gets full Node.js runtime execution. If your organisation uses Flowise — or if any of your development teams do — this is a patch-or-isolate decision you need to make today, not this quarter.

That MCP reference is not coincidental. This week SecurityWeek reported a structural vulnerability in Anthropic's Model Context Protocol that researchers describe as "by design." That phrase should stop you cold. This isn't a coding mistake someone can push a patch for. It's an architectural decision baked into the protocol itself, one that allows unsanitised commands to execute silently inside AI environments. The breadth of MCP adoption across the industry means you are likely running this exposure somewhere without knowing it. MITRE ATLAS frames this as ML supply chain compromise — and that framing is correct. The protocol layer is now an attack surface.

Speaking of supply chain, litellm — the Python library used by thousands of AI-integrated systems as a unified LLM API interface — was found compromised in version 1.82.8. A hidden .pth file executed arbitrary code on every Python interpreter startup. Silently. No trigger required. Any developer machine or AI pipeline pulling that version could be running attacker code right now. This is textbook ML supply chain compromise, and it's a direct reminder that your AI stack inherits every vulnerability of your Python dependency graph.

Now layer in what Palo Alto found in Google's Vertex AI. Researchers demonstrated that over-privileged AI agents on the platform could exfiltrate sensitive data and pivot into restricted cloud infrastructure. OWASP LLM08 — Excessive Agency — is the formal classification, but the strategic reality is simpler: AI workloads are being granted cloud permissions that no human operator would receive without scrutiny. Your IAM policies were not written with autonomous agents in mind. They need to be revisited with that assumption front and centre.

Finally, the "Comment and Control" research this week deserves attention from every CISO whose developers use AI coding agents. Claude Code, Gemini CLI, and GitHub Copilot Agents can all be manipulated through malicious instructions embedded in source code comments. An attacker poisons a repository, a developer opens it with their AI coding tool, and the agent executes the embedded instruction. The attack weaponises the agent's own helpfulness. Combined with the Cursor AI vulnerability — which chained prompt injection with a sandbox escape to achieve full shell access on developer machines — you have a credible, confirmed attack path from a poisoned code file to complete endpoint compromise.

The pattern this week is unmistakable. Agentic AI systems are being targeted at the protocol level, the supply chain level, and the interface level simultaneously. Attackers are not waiting for AI security to mature. They are mapping the trust assumptions baked into these systems and exploiting them now.

Three actions warrant immediate attention. Audit every AI agent deployment in your environment for excessive cloud and system permissions — apply least privilege as if these agents were untrusted external services, because under certain conditions they are. Verify the litellm version running in any AI pipeline and treat the compromise as a potential active incident until confirmed otherwise. And brief your development leads on Comment and Control attacks — the risk is not theoretical, and your developers are likely already using the affected tools.
